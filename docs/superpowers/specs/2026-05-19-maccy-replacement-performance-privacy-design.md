# Maccy Replacement Privacy And Performance Design

## Scope

This change makes the current Clipboard app credible as a local Maccy replacement on the user's real imported history dataset. The priority is not to clone every Maccy interaction. The priority is to prove performance against Maccy on the same Mac, make privacy and capture controls trustworthy, and add only the minimal QuickPanel management actions needed for daily replacement use.

The imported Maccy and Clipaste migration path is already considered validated by the user and is out of scope except where benchmark data needs to describe the resulting dataset.

## Goals

1. Produce repeatable same-machine benchmark evidence comparing Clipboard and Maccy on cold load, QuickPanel fetch/search, pasteboard write, large text handling, CPU, and memory.
2. Make Universal Clipboard, ignored pasteboard types, ignored source apps, pause/resume capture, and ignore-next-copy affect the running capture path without requiring restart.
3. Add QuickPanel management actions for delete selected item, pin/unpin selected item, clear unpinned items, and clear all history with confirmation.
4. Keep the implementation narrow: extend existing `ClipboardCore`, `ClipboardPlatform`, and `ClipboardApp` boundaries instead of replacing storage, search, or UI architecture.

## Non-Goals

- Full `LibraryWindow` implementation.
- Rich preview viewer, thumbnail cache, or custom group management UI.
- Plain-text-only paste or complete Maccy shortcut compatibility.
- CloudKit, multi-device app sync, App Store sandboxing, notarization, DMG, or Homebrew Cask packaging.
- Rewriting SQLite search or adding full-text indexing before benchmark evidence shows it is necessary.
- Reworking the already validated Maccy/Clipaste import pipeline.

## Current Context

The app already has background pasteboard capture through `ClipboardCaptureLoop`, `ClipboardMonitor`, and `ClipboardCaptureCoordinator`; SQLite-backed history and payload storage; QuickPanel search and type filters; paste transactions; settings pages; Maccy/Clipaste import; and verification scripts. Prior lightweight gap work deliberately left single-record management and privacy-policy runtime wiring for a later iteration.

The current `PrivacyPolicy` already models ignored pasteboard types, transient types, ignored app bundle IDs, and Universal Clipboard recording. The current settings UI exposes a Universal Clipboard toggle, but this design treats runtime capture-policy wiring as incomplete until tests prove settings changes affect the active capture path.

## Capture Control And Privacy Design

Add a small capture control layer in `ClipboardCore` named `CaptureControlService`. It owns the decision made after `ClipboardMonitor.poll()` returns a capture and before `ClipboardIngestService.ingest()` persists it.

The control layer answers:

- Is capture globally paused?
- Should this single pasteboard change be ignored because the user requested ignore-next-copy?
- Does the effective privacy policy allow this capture?

The effective policy is built from defaults plus user settings:

- `privacy.ignoreUniversalClipboard` maps to `recordsUniversalClipboard = false` when enabled.
- `privacy.ignoredPasteboardTypes` appends user-provided types to the standard ignored pasteboard types.
- `privacy.ignoredAppBundleIds` appends user-provided bundle IDs to ignored app bundle IDs.
- Standard concealed, transient, autogenerated, and known password-manager pasteboard types remain enabled by default.

Runtime settings changes must affect the running app without restart. `AppServices` should own the live control object and expose narrow methods used by settings and status-bar actions: pause capture, resume capture, ignore next copy, and refresh privacy settings.

Capture result should include a diagnostic reason for skipped captures so probes and tests can distinguish paused, ignore-next, Universal Clipboard ignored, pasteboard type ignored, and app ignored. A skipped capture should not be persisted and should not be treated as an error.

## QuickPanel Management Design

Add a small history mutation service named `HistoryMutationService` rather than letting QuickPanel know SQLite details. The service wraps `HistoryStore` plus `ClipboardPayloadStore`, and it uses store protocol extensions only where the current `HistoryStore` contract lacks the needed mutation.

Required operations:

- `deleteRecord(id:)`
- `togglePinned(id:)`
- `clearUnpinned()`
- `clearAll()`

Payload cleanup must reuse the existing payload-cleaning behavior so deleting a record does not leave orphan payload files. Pinned records must stay exempt from retention cleanup, consistent with existing `isPinned` and `retentionExempt` semantics.

QuickPanel should expose these actions in two ways:

- Keyboard shortcuts for daily speed: `Option+Delete` delete, `Option+P` pin/unpin, `Option+Command+Delete` clear unpinned, `Shift+Option+Command+Delete` clear all.
- A discoverable UI entry such as a compact action menu or row action area, so the feature is not hidden behind shortcuts.

Clear-all must require confirmation. Delete, pin/unpin, and clear-unpinned can complete directly but must update the footer with a success or failure message. After each mutation, QuickPanel refreshes while preserving a sensible selection.

## Benchmark Design

Add a benchmark path that can run on the user's current machine without modifying real history data. It should output both JSON and a concise human-readable summary.

The benchmark must first describe the dataset:

- Total records.
- Payload directory size.
- Counts by content type.
- Pinned count.
- Imported source markers when available.

Clipboard metrics:

- `cold_start_store_load_ms`: initialize the SQLite store and load the current index.
- `quickpanel_fetch_empty_query_ms`: fetch first 50 records with an empty query.
- `quickpanel_search_ms`: run several real queries, including short text, Chinese text, URL-like text, and source-app text.
- `pasteboard_write_ms`: write representative text, rich text, image, and file records to the system pasteboard with the app marker.
- `large_text_classify_ms`: classify 10MB JSON and 100MB log samples.
- `memory_rss_mb`: sample process memory at key phases where available.
- `cpu_sample`: sample CPU during capture/search/open loops where available.

Maccy comparison should be honest about automation limits. If a metric can be measured automatically, record it as automatic. If it requires UI or manual stopwatch-style interaction, record it as manual. If the metric is not comparable because the two apps expose different internals, record `not_comparable` with the reason.

Comparison result rules:

- `better`: Clipboard median is at least 20% lower than Maccy and p95 is not worse.
- `same`: Median difference is within 20%.
- `worse`: Clipboard median is more than 20% higher than Maccy, or visible failure/stutter occurs.
- `not_comparable`: boundaries or automation gaps prevent a fair comparison.

The benchmark must not claim that Clipboard is faster overall unless the relevant measured indicators support that claim.

## Data Flow

Capture path:

```text
ClipboardCaptureLoop
-> ClipboardMonitor.poll()
-> ClipboardCaptureCoordinator.captureLatestChange()
-> CaptureControlService.evaluate(capture)
-> allowed: ClipboardIngestService.ingest(capture)
-> skipped: return nil with diagnostic reason
```

QuickPanel management path:

```text
QuickPanelKeyCaptureView or action menu
-> QuickPanelState selected record
-> HistoryMutationService operation
-> HistoryStore / ClipboardPayloadStore mutation
-> QuickPanelState refresh
-> footer status or confirmation alert
```

Benchmark path:

```text
Scripts/benchmark-maccy-replacement.sh
-> ClipboardBenchmarkProbe executable
-> read-only dataset inspection
-> targeted store/query/pasteboard measurements
-> optional Maccy sampling
-> JSON + summary report
```

## Error Handling

Capture skips are normal control-flow events and should be diagnosable but not noisy. Storage mutation failures should leave the UI unchanged where possible and show a footer failure message. Clear-all confirmation prevents accidental destructive actions. Benchmark failures should produce a partial report with failed metric names, error messages, and whether any comparison is still valid.

## Tests And Verification

Unit tests should cover:

- Paused capture skips persistence.
- Resume capture allows persistence again.
- Ignore-next-copy skips exactly one external capture.
- Universal Clipboard ignore setting blocks captures with `com.apple.is-remote-clipboard` and allows them when disabled.
- User ignored pasteboard types and app bundle IDs affect the policy.
- Delete selected record removes the record and cleans payload.
- Pin/unpin updates record metadata and retention exemption behavior.
- Clear unpinned preserves pinned records.
- Clear all removes records and payloads after confirmation path calls the mutation service.
- Benchmark classification logic produces `better`, `same`, `worse`, and `not_comparable` correctly.

Integration or script verification should cover:

- `swift test`.
- `Scripts/verify.sh`.
- Benchmark script against the current imported dataset.
- Extended `ClipboardManualProbe` commands proving pause/resume, ignore-next-copy, and Universal Clipboard ignore behavior where possible.
- App bundle build with existing signing script.

Manual acceptance should cover:

- Toggle pause/resume from the visible app surface and confirm history growth behavior.
- Trigger ignore-next-copy and confirm only the next copy is skipped.
- Use QuickPanel shortcuts for delete and pin/unpin.
- Clear unpinned and clear all with expected confirmation behavior.
- Review benchmark summary and raw JSON before making any performance claim.

## Acceptance Criteria

The change is complete only when:

1. Current branch is verified with `swift test`, `Scripts/verify.sh`, and app bundle build.
2. Runtime privacy settings affect active capture without restart.
3. Pause/resume and ignore-next-copy have automated or probe-based evidence.
4. QuickPanel can delete, pin/unpin, clear unpinned, and clear all with payload cleanup and refreshed selection.
5. Benchmark output exists as JSON and readable summary for the user's current imported history dataset.
6. Any claim about being faster than Maccy is phrased per metric using `better`, `same`, `worse`, or `not_comparable`.

## Risks

- Maccy UI metrics may not be fully automatable. The report must label comparison confidence instead of over-claiming.
- Current in-memory index and substring filtering may be enough for the user's dataset but may not scale indefinitely. Benchmark evidence decides whether FTS or a new index becomes a later task.
- Universal Clipboard can be hard to trigger automatically. Unit tests should simulate the pasteboard type, while manual acceptance validates the real device path.
- Keyboard shortcuts may conflict with text input or system behavior. The implementation should preserve existing search focus and only intercept the specified shortcuts inside QuickPanel.
